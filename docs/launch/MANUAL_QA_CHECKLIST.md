# 결리포트 Manual QA Checklist

## 1. 목적

이 문서는 출시 전 브라우저에서 직접 확인해야 할 수동 QA 절차를 정리한다.

- 자동 테스트가 놓칠 수 있는 UI, 문구, 라우팅, 결제 비활성 상태 문제를 확인한다.
- 홈에서 리포트 생성, 결과 미리보기, 정책 페이지까지 실제 사용자 흐름을 점검한다.
- 테스트 결과와 이슈를 같은 형식으로 기록해 출시 판단에 활용한다.

## 2. QA 전제

- 결리포트는 현재 출시 전 단계의 앱이다.
- 실제 결제는 활성화되어 있지 않다.
- 이용약관, 개인정보 처리방침, 환불 안내 페이지는 placeholder 성격의 초안이다.
- 리포트 생성 미리보기는 사용할 수 있다.
- 수동 QA는 법무 검토나 결제 운영 검토를 대체하지 않는다.

## 3. 실행 전 준비

- 로컬 확인이면 개발 서버를 실행한다.
- QA 전에 release check를 먼저 실행한다.
- 데스크톱과 모바일 viewport를 모두 확인한다.
- 확인한 브라우저, 기기, 날짜를 기록한다.

```bash
pnpm release:check
```

## 4. 홈 화면 QA

- [ ] `/` 경로가 정상적으로 열린다.
- [ ] `샘플 리포트 생성하기` CTA가 `/report/new`로 연결된다.
- [ ] 문의 이메일이 표시되어 있다면 읽기 쉽고 복사 가능하다.
- [ ] 정책 페이지 링크가 있다면 `/terms`, `/privacy`, `/refund`로 이동할 수 있다.
- [ ] 실제 유료 구매가 가능한 것처럼 오해될 문구가 없다.

## 5. 리포트 생성 화면 QA

- [ ] `/report/new` 경로가 정상적으로 열린다.
- [ ] 출생정보와 MBTI 입력 영역이 보인다.
- [ ] 제출 버튼의 기본 상태가 명확하다.
- [ ] 제출 중 loading 상태가 표시된다.
- [ ] 생성 성공 상태가 표시된다.
- [ ] 오류 상태가 사용자에게 차분하게 안내된다.

## 6. 입력 검증 QA

- [ ] 양력 생년월일과 출생시간을 입력해 리포트가 생성된다.
- [ ] 출생시간을 모르는 경우 또는 비워 둔 경우의 동작을 확인한다.
- [ ] MBTI를 선택했을 때 관련 리포트 내용이 표시된다.
- [ ] MBTI를 생략했을 때도 흐름이 중단되지 않는다.
- [ ] 필수 생년월일이 빠졌을 때 검증 안내가 표시된다.
- [ ] 잘못되었거나 incomplete한 입력값에서 오류 안내와 재시도 경로가 보인다.

## 7. 리포트 결과/미리보기 QA

- [ ] 리포트 섹션들이 화면에 렌더링된다.
- [ ] 일주, 오행, 십성, 신살 등 핵심 사주 용어가 읽기 좋게 표시된다.
- [ ] MBTI를 입력한 경우 MBTI 연계 내용이 보인다.
- [ ] 미리보기 또는 개발용 preview mode 안내가 보인다.
- [ ] 결정론적이거나 과도하게 단정적인 주장처럼 보이는 문구가 없다.
- [ ] 전체 리포트 잠금 또는 gating 관련 문구가 실제 결제 가능 상태로 오해되지 않는다.

## 8. 정책 페이지 QA

- [ ] `/terms` 경로가 열린다.
- [ ] `/privacy` 경로가 열린다.
- [ ] `/refund` 경로가 열린다.
- [ ] 각 페이지에 출시 전 초안 안내가 있다.
- [ ] 각 페이지에 `official@dvem.ai` 문의 이메일이 보인다.
- [ ] 각 페이지의 홈으로 돌아가기 링크가 `/`로 이동한다.

## 9. 결제 비활성 QA

- [ ] 실제 결제 provider 흐름이 시작되지 않는다.
- [ ] 카드나 결제 정보 입력 UI가 나타나지 않는다.
- [ ] paid unlock이 실제로 활성화된 것처럼 표시되지 않는다.
- [ ] 유료 출시 전에는 결제 비활성 상태가 명확하게 유지된다.

## 10. 모바일 QA

- [ ] 360px 너비 viewport에서 홈 화면이 읽기 좋다.
- [ ] 360px 너비 viewport에서 가로 overflow가 없다.
- [ ] 모바일에서 리포트 생성 form을 사용할 수 있다.
- [ ] 모바일에서 리포트 카드와 긴 목록을 읽을 수 있다.
- [ ] 모바일에서 정책 페이지 문구와 링크를 사용할 수 있다.

## 11. 접근성/문구 QA

- [ ] 주요 heading 계층이 읽기 쉽다.
- [ ] 버튼과 링크의 목적이 명확하다.
- [ ] 한국어 문구가 깨지거나 어색하게 줄바꿈되지 않는다.
- [ ] 문의 이메일을 쉽게 찾을 수 있다.
- [ ] 리포트 해석이 참고자료임을 알리는 주의 문구가 보인다.

## 12. 오류/복구 QA

- [ ] 네트워크 또는 API 오류를 시뮬레이션할 수 있으면 오류 안내를 확인한다.
- [ ] 검증 오류 문구가 사용자가 이해할 수 있는 형태로 보인다.
- [ ] 실패 후 다시 시도할 수 있는 흐름이 있다.
- [ ] 오류 이후 사용자가 화면에 갇히지 않는다.

## 14. 출시 전 차단 조건

- release check가 실패한다.
- 리포트 생성 흐름이 동작하지 않는다.
- 정책 페이지 라우트에 접근할 수 없다.
- 실제 결제가 가능한 것처럼 오해될 결제 문구가 있다.
- 개인정보 또는 저장소 경계가 불명확하다.
- 모바일 화면에서 주요 흐름을 사용할 수 없다.
- 문의 이메일을 찾을 수 없다.
